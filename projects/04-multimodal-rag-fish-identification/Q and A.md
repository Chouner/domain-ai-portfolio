
**为什么用CLIP做视觉召回，而不是直接让Qwen-VL识别？**

Qwen-VL是生成模型，每次调用要推理一遍，慢且贵。如果有110个物种，让它对每个物种都比对一遍不现实。

CLIP是向量模型，一张图秒出一个512维向量，向量之间做余弦相似度是纯数学运算，毫秒级。所以CLIP做的事是**粗筛**——快速从110个里圈出10个候选，Qwen-VL只需要在这10个里做精细判断。

这是一个经典的**召回-精排**两阶段架构，搜索引擎、推荐系统都是这个思路，便宜的模型大范围粗筛，贵的模型小范围精排。

---

**为什么视觉描述和文本RAG分开，不直接让Qwen-VL输出物种名？**

直接问Qwen-VL"这是什么鱼"有两个问题：

第一，它可能根本不认识深海鱼类，会幻觉。物种是南海深海捕捞的，不是常见鱼类，通用视觉模型见过的训练数据里这些物种很少。

第二，即使它认识，没办法知道它为什么这么判断，不可追溯，不可审计，出错了也不知道从哪改。

把它拆成两步——Qwen-VL只做"我能看到什么"（体形、鳍型、发光器），RAG做"这些特征对应哪个物种"——每一步都有输入输出，每一步都可以单独验证和改进。

这是你方案文档里写的核心设计原则：**可追溯、可审计**。

---

**为什么embedding用text-embedding-v4而不是直接用CLIP的文本向量？**

CLIP的文本向量是为了和图像向量对齐设计的，在图文相似度任务上很强，但在纯文本语义检索上不如专门的文本embedding模型，尤其是中文专业术语（鳍条数、发光器分布这类描述）。

text-embedding-v4是专门优化过中文语义的，对"体延长侧扁臀鳍基底较长"这类专业描述的向量表示更准确。

**测试集如何整理？**

测试图抽取并结构化

每张测试图需要记录这些信息，存成 `test_set_log.csv`：

```
image_file, species_id, chinese_name, quality, angle, in_kb
```

`angle` 是拍摄角度：侧面/正面/局部/背面。这个字段很重要，后面分析错误时你需要知道"是正面图准确率低还是侧面图低"。

`in_kb` 标记，固定填0，表示这张图没有进知识库。

每个物种的测试图里，要有至少1张低质量或中等质量的图，防止覆盖不到实际情况
---

**结果如何量化？**

分三个层次，从简单到复杂。

**第一层：准确率指标**

跑完89张记录结果：

```
Top-1准确率：系统第一推荐 = 正确物种 的比例
Top-3命中率：正确物种在前三推荐内 的比例
科级准确率：推荐物种和正确物种同科 的比例
```

科级准确率很重要，因为就算Top-1错了，如果科都对，说明系统方向没偏，只是种间区分不够细。这个指标能帮你解释"为什么80%的Top-1准确率其实已经很有价值"。

**第二层：稳定性指标**

从89张里随机抽10张，每张跑3次，记录3次结果：

```
跨轮次一致性：3次推荐完全一致的图片数 / 10
```

记录方式：

```
image_file, run1_pred, run2_pred, run3_pred, consistent
```

**第三层：相似种压力测试**

专门针对眶灯鱼属和巴氏眶灯鱼这两个相似种，各10张测试图共20张，单独统计：

```
相似种混淆率：两者互相错认的次数 / 20
```

---

**如何记录结果和评估结果？**

跑完之后 `test_results.csv` 长这样：

```
image_file, true_species, pred_top1, pred_top3, top1_correct, top3_correct, family_correct, confidence, quality, angle, error_reason
```

`error_reason` 只在top1_correct=0时填，分几类：知识库内容不足、图片质量差、相似种混淆、其他。

填完之后直接算出所有指标，也能做错误分析——按error_reason分组，看哪类错误最多，对应改进方向。
